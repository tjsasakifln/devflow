## Story

so that **the preview tier is fully functional**.

## Scope

**IN:** `tests review` (test-plan.md vs implementação, gaps de coverage), `actions generate` (GitHub Actions workflow do project config), `drift check` (requirements.md vs implementation-log.jsonl).
**OUT:** Batch 3.

**Complexity:** M (3 comandos, `drift check` mais complexo — diff semântico entre docs)

## Acceptance Criteria

1. `devflow tests review` — verifica test-plan.md contra implementação, detecta gaps de coverage
2. `devflow actions generate` — gera GitHub Actions workflow a partir do project config
3. `devflow drift check` — compara requirements.md vs implementation-log.jsonl, detecta divergências

## Integration Verification

1. IV1: Cada comando tem teste CLI (pipe-safe JSON)
2. IV2: Actions generate produz YAML válido e funcional
3. IV3: Drift check detecta AC implementada mas não documentada (e vice-versa)

## Dev Notes

- `drift check` é o mais complexo: diff semântico entre docs e código. Usar heurísticas com flag de controle
- `actions generate` usa template base — não reinventar, usar o que já está documentado no README
- `tests review` complementa coverage (Story 4.4) — foca em documentação vs realidade, não em %

## Risks

Médio — `drift check` usa heurísticas para comparar docs vs código. Pode gerar falsos positivos. Mitigação: flag `--strict` vs `--heuristic`.
