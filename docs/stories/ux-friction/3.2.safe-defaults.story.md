**Status:** InProgress

**Audit trace:** Achados #8.1 (paradoxo: usuário burla todos os gates), #6.1 (25 checks mal categorizados), #6.4 (modos de execução sem explicação)

## Problem Statement

Usuário preguiçoso usa `--mode local` + `riskTolerance relaxed` e consegue burlar TODOS os gates: checks viram advisory, C12 é waivable, artifacts falsos passam. O sistema não detecta nem avisa quando o usuário está sistematicamente contornando as proteções.

## Scope

**IN:**
- Bypass pattern detection engine tracking 3 metrics across features
- Consolidated warning display when patterns detected
- Quality Debt counter in `devflow status`
- `--require-justification` flag for config hardening
- Doctor integration for bypass health check
- Local-only tracking (no telemetry)

**OUT:**
- Blocking/refusing commands based on bypass detection (warn only)
- Remote telemetry or analytics
- Automatic config changes without user consent
- Per-team or per-organization aggregate metrics
- Historical trend analysis beyond simple pattern detection

## User Story

AS a Devflow system
I WANT to detect when a user is systematically bypassing quality gates
SO THAT I can warn them and suggest stricter settings before bad code reaches production

## Acceptance Criteria

1. **Desperation detection**: Track bypass patterns across features:
   - 3+ features with `riskTolerance relaxed` in sequence → warning
   - 2+ `gatekeep --approve` with same actor without adversarial review → warning
   - `feature complete` with <50% pass rate 3+ times → warning
2. When desperation detected, show consolidated warning with current settings and suggested fix
3. `devflow status` shows "Quality Debt" counter: "⚠️ 3 features with bypassed gates"
4. `devflow config set riskTolerance strict --require-justification` forces 1-line reason
5. Default `riskTolerance` stays `moderate` (current default is correct)
6. `devflow doctor` includes bypass pattern detection in health report

### Implementation Summary

Created the bypass pattern detection and warning system (Story 3.2) with:

1. **Pattern Detection Engine** (`src/kernel/tracking/bypass-detector.ts`):
   - Records bypass events to `.devflow/audits/bypass-log.jsonl` (local only, no telemetry)
   - Detects 3 patterns: riskTolerance-relaxed (3+ features), gatekeep-same-actor (2+ features), feature-complete-low-pass-rate (3+ features)
   - Generates consolidated warning with current settings and suggested fixes
   - Quality Debt counter (unique features with bypass events)

2. **Gatekeep Integration** (`src/commands/gatekeep.ts`):
   - Records bypass event when same actor approves without adversarial review

3. **Feature Complete Integration** (`src/commands/feature-complete.ts`):
   - Records bypass event when pass rate < 50%

4. **Status Display** (`src/commands/status.ts`):
   - Shows quality debt counter in both human and JSON output
   - "⚠️  N feature(s) with bypassed gates"

5. **Doctor Integration** (`src/commands/doctor.ts`):
   - Check #19: Bypass pattern detection — FAIL with remediation when patterns detected, PASS when healthy

6. **Config Hardening** (`src/cli/index.ts`):
   - `--require-justification` flag for `devflow config set`
   - Forces 1-line reason for riskTolerance strict
   - Logs config changes to bypass log

### Decisions Made

| Decision | Rationale | Alternatives |
|----------|-----------|-------------|
| Store in `.devflow/audits/bypass-log.jsonl` | Follows existing audit pattern (.devflow/audits/gatekeep-log.jsonl) | Store in state.json would mix concerns |
| Unique feature counting | Prevents false positives from repeated events on same feature | Raw entry count would over-report |
| Warn only, never block | Per AC: "Does NOT block — warns only (advisory, not punitive)" | Blocking would violate story scope |
| Pattern 2 counts unique features, not entries | A single feature with 2 same-actor approvals is 1 event, not pattern trigger. Need 2 different features | Different interpretation could trigger on single feature |
