# Story 2.1: Templates Separados por Contexto (Greenfield vs Brownfield)

**Epic:** UX Friction Reduction — Devflow Onboarding
**Sprint:** 2 (Médio Prazo)
**Priority:** P0 — Bloqueante
**Effort:** 5 pontos
**Status:** Done

## Problem Statement

Templates de artifacts (requirements.md 15 seções, roadmap.md 13 seções) foram projetados para o ecossistema Reversa (engenharia reversa de legado). Seções como "Contexto a partir do Legado", "Invariantes de Domínio", "Delta Arquitetural", "Delta de Dados" não fazem sentido para projetos greenfield. Usuário preenche com placeholder ou "n/a" — frustração e baixa qualidade.

## User Story

AS a developer starting a new (greenfield) project
I WANT templates that only ask for what's relevant to new projects
SO THAT I can fill them quickly without ignoring irrelevant sections

## Acceptance Criteria

1. Two template variants for each artifact:
   - `requirements-template.greenfield.md` — 10 seções (sem legado, sem delta)
   - `requirements-template.brownfield.md` — 11 seções (completo, atual)
   - `roadmap-template.greenfield.md` — 5 seções (Descrição Técnica, Decisões Técnicas, Estrutura de Arquivos, Dependências Externas, Riscos)
   - `roadmap-template.brownfield.md` — 11 seções (completo, atual)
2. `feature new` auto-selects template based on project type detection:
   - Greenfield (no src/, new repo) → greenfield templates
   - Brownfield (existing codebase) → brownfield templates
3. `--template greenfield|brownfield` flag overrides auto-detection
4. DoD checks adapt expected section count based on template variant
5. Existing brownfield templates preserved as-is (backward compatible)
6. Greenfield templates use simplified language: "O que vai ser construído" instead of "Delta Arquitetural"

### Key Findings

**REQ-001 (high):** `generatePrefilledRequirements()` em `feature.ts` sempre gera requisicoes no formato brownfield (15 secoes, incluindo "Invariantes de Dominio", "Casos Extremos", "Restricoes Tecnicas", "Escopo Negativo", "Riscos de Manutencao") independentemente do project type detectado. A variante detectada (greenfield/brownfield) afeta apenas o rotulo de exibicao, nao o conteudo gerado.

**REQ-002 (high):** O arquivo `requirements-template.greenfield.md` contem 9 secoes, mas a AC 1 especifica 10 secoes. O validador greenfield (`REQUIREMENTS_GREENFIELD_SECTIONS`) define 10 secoes com nomes diferentes do template.

**TEST-001 (medium):** Nenhum teste unitario para `detectProjectType()`, `validateRequirementsVariant()` ou `validateRoadmapVariant()`. Funcoes de validacao variant-aware sem cobertura (exempted por YOLO mode).

**MNT-001 (low):** Nomes de secoes nos templates (`templates/`) diferem completamente dos nomes nos validadores (`structural.ts`). Templates usam "Resumo executivo" enquanto validadores esperam "Descricao Funcional".
