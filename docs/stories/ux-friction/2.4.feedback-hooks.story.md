**Epic:** UX Friction Reduction — Devflow Onboarding

**Priority:** P1 — Alta
**Effort:** 3 pontos
**Status:** Done
**Depends on:** Story 1.2 (stage-aware-dod) — state detection for health summary
**Audit trace:** Achados #6 (fadiga de gates), #8.2 (ciclo de feedback só no fim), #2.3 (next assume conhecimento prévio)

## Problem Statement

Usuário só descobre que fez algo errado no `feature complete` (25 checks de uma vez). Nesse ponto, já investiu horas. Feedback deveria vir PROATIVAMENTE após cada comando relevante, não apenas quando o usuário lembra de rodar `doctor` ou `next`.

## Scope

**IN:**
- Pre-condition warnings for 4 high-impact commands: `feature prompt`, `feature complete`, `feature new`, `gatekeep`
- Mini health summary after state-transitioning commands
- Centralized hook registration system in `src/main.ts`
- Per-feature warning snooze persistence
- `--no-warnings` flag to suppress all warnings

**OUT:**
- Auto-fixing issues (warnings only — no automatic state transitions)
- Telemetry or analytics on warning interactions
- Warnings for commands outside the feature lifecycle (doctor, status, config, install)
- Multi-language warning messages (English only for v1)
- Integration with external notification systems (Slack, email)

## User Story

AS a developer moving through the feature lifecycle
I WANT proactive warnings when I'm about to make a mistake
SO THAT I can fix issues early instead of failing 25 checks at the end

## Acceptance Criteria

1. After each state-transitioning command, show a mini health summary with pass/warn counts
2. Specific pre-condition warnings:
   - `feature prompt` without complete artifacts → warn about placeholders, offer to complete first
   - `feature complete` with <50% artifacts filled → warn most checks will fail, confirm continue
   - `feature new` in brownfield without `discover` → suggest running discover first
   - `gatekeep` without `adversarial-review` → suggest running review first
3. Warnings are advisory (don't block) but visible (yellow + icon)
4. `--no-warnings` flag suppresses all proactive warnings
5. Warning logic is centralized in a hook system, not duplicated per command
6. Each warning has a `snooze` option per feature
